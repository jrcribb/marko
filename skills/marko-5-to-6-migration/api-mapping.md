# Marko 5 → Marko 6 API Mapping

Exhaustive translation reference. Marko 5 docs: <https://v5.markojs.com/docs/>; Marko 6 docs: <https://markojs.com/docs/> (append `.md` to any page URL for markdown). When unsure about Marko 6 behavior, check the docs — do not guess.

## Template syntax

| Marko 5                                                                                                     | Marko 6                                                                                                   | Notes                                                                                                                                                                                                                        |
| ----------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `${expr}`                                                                                                   | `${expr}`                                                                                                 | unchanged (escaped)                                                                                                                                                                                                          |
| `$!{expr}`                                                                                                  | `$!{expr}`                                                                                                | unchanged (unescaped; same XSS caution)                                                                                                                                                                                      |
| `$ statement;` / `$ { ... }` scriptlet                                                                      | `<const/x=...>`, `<let/x=...>`, or an IIFE `<const/x=(() => { ... })()>` for multi-statement derivations  | Scriptlets are Class-only. Reassigned-during-render vars usually become a single `<const>` computing the final value                                                                                                         |
| `static ...` / `static { }`                                                                                 | `static ...` / `static { }`                                                                               | unchanged (module scope, runs once)                                                                                                                                                                                          |
| env guards (`typeof window` checks, browser-only `require`) and bundling-trick `component-browser.js` files | `server` / `client` statements: `client import "./browser-only"`, `server { ... }`                        | backported to Marko 5.38+ and neutral for interop classification — adopt in Class files **before** migrating; a `component-browser.js` that only exists to keep code out of the server bundle is a glorified `client import` |
| `class { ... }` block                                                                                       | delete; migrate members per "Component class API" below                                                   |                                                                                                                                                                                                                              |
| `style { ... }` / `style.scss { ... }` block                                                                | `<style>` / `<style.scss>` tag                                                                            | Marko 6 `<style>` additionally supports CSS Modules via a tag variable (`<style/styles>` → `styles.foo`) and reactive `${...}` values (compiled to CSS custom properties)                                                    |
| `import x from "<my-tag>"`                                                                                  | unchanged                                                                                                 | Marko 6 adds lazy variant: `import X from "<my-tag>" with { load: "visible#sel" }`                                                                                                                                           |
| `export ...`                                                                                                | unchanged                                                                                                 |                                                                                                                                                                                                                              |
| attribute values `attr=expr`                                                                                | unchanged                                                                                                 | both parse values as JS expressions; in 6 an unparenthesized `>` inside a value is a parse hazard — wrap: `x=(a > b)`                                                                                                        |
| spread `...expr`                                                                                            | unchanged                                                                                                 |                                                                                                                                                                                                                              |
| `class=` / `style=` object & array values                                                                   | unchanged                                                                                                 |                                                                                                                                                                                                                              |
| shorthand `.class` / `#id` (incl. `.icon-${name}`)                                                          | unchanged                                                                                                 |                                                                                                                                                                                                                              |
| attribute arguments `attr(...)` (e.g. `on-click("m")`, `no-update-if(x)`, `body-only-if(x)`)                | removed — the parenthesized-attribute syntax is Class-only                                                | see rows for each specific attribute                                                                                                                                                                                         |
| `body-only-if(cond, tagName?)`                                                                              | conditional parent tag: `<${!cond && "a"} href=...>content</>`                                            | note the inversion: `body-only-if` drops the wrapper when `cond` is truthy; a falsy dynamic tag name renders content only                                                                                                    |
| `key="name"` (element/component reference)                                                                  | tag variable: `<div/name>`                                                                                | see "Refs" below                                                                                                                                                                                                             |
| `key="name[]"` (repeated)                                                                                   | tag variable inside `<for>`; iterate the hoisted var (`[...name]`)                                        |                                                                                                                                                                                                                              |
| `key=expr` on children of `<for>`                                                                           | `by=` attribute on `<for>` itself (`by="id"` or `by=(item) => item.id`)                                   |                                                                                                                                                                                                                              |
| `on-event("method", ...args)` / `on-event(fn)`                                                              | `onEvent() { ... }` or `onEvent=fn` function attributes                                                   | curried args become closure captures; `on-` dash-case still exists in 6 for case-preserving DOM event names (`on-DblClick`)                                                                                                  |
| `once-event(...)`                                                                                           | no equivalent attribute                                                                                   | use a conditional handler (`onClick=!done && handler`) or `el().addEventListener(..., { once: true })` in a `<script>`                                                                                                       |
| `no-update`, `no-update-if(...)`, `no-update-body`, `no-update-body-if(...)`, `attr:no-update`              | removed; updates are granular so most uses are obsolete                                                   | if the intent was "Marko must not touch third-party-owned DOM", give the library its own container element managed via `<lifecycle>`; if the intent was "keep state/DOM alive while hidden", use `<show>`                    |
| `attr:scoped` (`id:scoped`, `for:scoped`, ...)                                                              | `<id/name>` tag + plain attributes (`id=name`, `for=name`)                                                |                                                                                                                                                                                                                              |
| `<tag\|params\|>` body parameters                                                                           | unchanged                                                                                                 | Marko 6 adds tag _arguments_: `<${input.content}(a, b)/>` ↔ `<my-tag\|a, b\|>`                                                                                                                                               |
| `<@section>` attribute tags                                                                                 | unchanged syntax; consumed as `input.section.content` (was `.renderBody`); repeated tags iterable in both | Marko 6: attribute tags inside `<if>`/`<for>` apply conditionally/repeatedly; typed with `Marko.AttrTag<T>`                                                                                                                  |
| `<${dynamic}>` dynamic tags                                                                                 | unchanged                                                                                                 | in 6 a string always renders a _native_ tag; pass the component reference itself for custom tags                                                                                                                             |
| `<${input.renderBody}/>`                                                                                    | `<${input.content}/>`                                                                                     | `content` also accepts attributes/arguments to pass tag parameters back                                                                                                                                                      |
| PascalCase variables as tags `<MyTag/>`                                                                     | unchanged                                                                                                 |                                                                                                                                                                                                                              |
| HTML/JS comments                                                                                            | unchanged (stripped)                                                                                      | literal comment: `<html-comment>` in both                                                                                                                                                                                    |
| concise mode                                                                                                | unchanged in spirit                                                                                       | see <https://markojs.com/docs/reference/concise-syntax.md> for 6 specifics                                                                                                                                                   |
| `<!-- use class -->` / `<!-- use tags -->`                                                                  | only meaningful under Marko 5 interop                                                                     | delete after the final flip to `marko@6`                                                                                                                                                                                     |

New in Marko 6 (no 5 equivalent — reach for these while converting):

- `:=` bound attributes / change handlers (`value:=state`, refining form `value:parseFloat:=num`) — the controllable pattern.
- Tag variables on custom tags via `<return>`; on native tags they expose a DOM-node getter.
- `<show>` (toggle visibility, keep state/DOM), `<id>`, `<log>`, `<debug>`, `<define>`, `<try>`, `<lifecycle>`.
- Controllable native attributes: `<input valueChange= checkedChange= checkedValueChange=>`, `<select value= valueChange=>`, `<textarea value=>`, `<details openChange=>`, `<dialog openChange=>`.
- `$signal` (an `AbortSignal` for cleanup in any expression) and `$global`.
- Lazy tag imports `with { load: "visible|idle|media|on*|render" }`.

## Core tags

| Marko 5                                     | Marko 6                                                                   | Notes                                                                                                                                              |
| ------------------------------------------- | ------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| `<if(cond)>` / `<else-if(cond)>` / `<else>` | `<if=cond>` / `<else if=cond>` / `<else>`                                 | argument syntax → `value` attribute; `else-if` tag name → `else` with `if=` attribute                                                              |
| `<for\|item, index, list\| of=list>`        | `<for\|item, index\| of=list>`                                            | third `list` param dropped; add `by=` for keyed diffing                                                                                            |
| `<for\|key, value\| in=obj>`                | unchanged                                                                 |                                                                                                                                                    |
| `<for\|i\| from=x to=y step=z>`             | unchanged; 6 adds exclusive `until=`                                      |                                                                                                                                                    |
| `<while(cond)>`                             | removed                                                                   | restructure into `<for>` over a computed list/range                                                                                                |
| `<macro\|params\| name="foo">` + `<foo>`    | `<define/Foo\|input\|>` + `<Foo>`                                         | params become `input` properties/content                                                                                                           |
| `<await(promise)>` + `<@then\|res\|>`       | `<await\|res\|=promise>`                                                  | body is the "then"                                                                                                                                 |
| `<await>` `<@placeholder>`                  | `<try>` ancestor with `<@placeholder>`                                    | placeholder now lives on `<try>`, can wrap several awaits                                                                                          |
| `<await>` `<@catch\|err\|>`                 | `<try>` ancestor with `<@catch\|err\|>`                                   |                                                                                                                                                    |
| `<await client-reorder>`                    | automatic                                                                 | a `<try>` `@placeholder` opts into out-of-order streaming; no flag                                                                                 |
| `<await timeout= show-after= name=>`        | removed                                                                   | implement timeouts on the promise itself (`AbortSignal.timeout` / `Promise.race`)                                                                  |
| `<await-reorderer/>`                        | removed                                                                   | already implicitly injected by Marko 5 — explicit usages are redundant; delete unconditionally                                                     |
| `<include-text('./f.txt')>`                 | bundler raw import (`import text from "./f.txt?raw"` in Vite) + `${text}` |                                                                                                                                                    |
| `<include-html('./f.html')>`                | raw import + `$!{html}`                                                   | audit for XSS while converting                                                                                                                     |
| `<init-components/>`                        | removed                                                                   | already implicitly injected by Marko 5 — explicit usages are redundant; delete unconditionally (Marko 6 emits hydration boilerplate automatically) |
| `<module-code>`                             | removed                                                                   | library codegen hook; needs bespoke redesign (compiler hooks)                                                                                      |
| `<html-comment>`                            | unchanged                                                                 | 6 version also exposes a comment-node tag variable                                                                                                 |
| `<script>` (plain HTML script)              | `<html-script>`                                                           | in 6, `<script>` is the _effect_ tag (runs in browser, re-runs when referenced state changes)                                                      |
| `<style>` (plain HTML style)                | `<html-style>`                                                            | in 6, `<style>` extracts to the CSS bundle                                                                                                         |
| `<context>` (npm package)                   | no equivalent                                                             | pass data through `input` or `$global`                                                                                                             |

If the app used `@marko/tags-api-preview` (Tags API on Marko 5 before 6 shipped): remove that package; `<let>`/`<const>`/`<return>`/`<define>` carry over, `<effect>` → `<script>`, `<attrs>` → destructuring `<const/{...}=input>` (the Marko 6 compiler auto-fixes these two), and `<get>`/`<set>` context tags must be redesigned onto `input`/`$global`.

## Component class API

The class (inline `class {}`, `component.js`, `component-browser.js`, or plain-object form) disappears entirely; each member maps to template-level constructs in the one `.marko` file.

### State

| Marko 5                                                     | Marko 6                                                                                                     |
| ----------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| `onCreate() { this.state = { a: 1, b: 2 } }`                | `<let/a=1>` `<let/b=2>` — one `<let>` per independently-changing key                                        |
| read `state.a` / `this.state.a`                             | `a`                                                                                                         |
| `this.state.a = v` / `setState("a", v)` / `setState({ a })` | `a = v` (assignment is the reactive trigger)                                                                |
| `setStateDirty("list")` after mutation                      | intentionally no equivalent — make the update immutable: `list = [...list, item]`                           |
| `replaceState(newState)`                                    | assign each `<let>`                                                                                         |
| `shouldUpdate(input, state)`                                | none — updates are per-expression; if work must be gated, gate it in the deriving `<const>`/handler         |
| `update_<name>(new, old)` partial-update handlers           | none — granular updates make them obsolete; side effects on change move to `<script>` reading that variable |
| `this.forceUpdate()`                                        | none needed; state assignment schedules the update                                                          |
| `this.update()` (sync flush)                                | none in-template (updates batch after a microtask); tests should await a tick                               |

Serialization in 6 is **need-based, not blanket**: the compiler computes what must cross to the browser from what client-side code references (handlers, `<script>` bodies, reactive expressions, and their intersections) and serializes only that. Values that never reach client code aren't serialized at all. The practical rule while converting: anything browser code touches must be serializable (no class instances, DOM nodes, or server-only resources there) — see <https://markojs.com/docs/explanation/serializable-state.md>.

### Derived data & input

| Marko 5                                                                   | Marko 6                                                  |
| ------------------------------------------------------------------------- | -------------------------------------------------------- |
| `$ const full = a + b;`                                                   | `<const/full=a + b>`                                     |
| `onInput(input) { input.x = normalize(input.x) }` (mutation anti-pattern) | `<const/x=normalize(input.x)>`                           |
| `onInput` storing input into state ("controllable" components)            | `<let/x:=input.x>` — controllable let; see `patterns.md` |
| `this.input` / template `input`                                           | `input` (unchanged)                                      |
| template `state` variable                                                 | the individual tag variables                             |
| template `component` variable                                             | none — every use maps to a specific pattern below        |
| template `out` variable                                                   | none; `out.global` → `$global`                           |

A Marko 5 getter (`get isFull() { return a + b >= max }`) recomputes on every read, so a handler that assigns `a` then reads `this.isFull` sees the new result. A `<const>` does not: within one handler it still holds its pre-assignment value until the update flushes, while the raw `<let>` reads back the value you just assigned. So after `moves = next`, `moves >= max` is current but a derived `<const/isFull=(moves >= max)>` is not — read the `<let>`s you just assigned (or recompute from them) inside the handler, never the derived `<const>`.

### Lifecycle

| Marko 5                                                                  | Marko 6                                                                                                                    |
| ------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------- |
| `onMount() { ... }`                                                      | `<script>` (first run = mounted, browser-only) or `<lifecycle onMount() { ... }/>`                                         |
| `onUpdate()`                                                             | `<script>` re-runs when any referenced tag variable/param changes; or `<lifecycle onUpdate>` (re-runs on its dependencies) |
| `onDestroy()`                                                            | `$signal.onabort = () => {...}` inside `<script>`; or `<lifecycle onDestroy>`                                              |
| `onRender(out)`                                                          | usually deleted — it was mostly used for derivations (→ `<const>`)                                                         |
| `onCreate(input, out)`                                                   | initial values of `<let>` tags                                                                                             |
| lifecycle _events_ (`this.on("update", fn)`, `this.once("destroy", fn)`) | same replacements as the methods                                                                                           |
| `on-attach(fn)` on an element                                            | `<script>` or `<lifecycle onMount>` — both run with the element in the DOM                                                 |
| `on-detach(fn)` on an element                                            | no equivalent — nothing can delay removal (see below)                                                                      |

Prefer plain state and `<script>`; use `<lifecycle>` only to host imperative third-party instances (maps, charts) where a stable `this` across `onMount`/`onUpdate`/`onDestroy` is genuinely needed.

Exit animations need rework. Marko 5's `on-detach` handler could hold a node in the DOM (`event.preventDefault()`, then `event.detach()` once the animation finished); nothing in Marko 6 can delay removal — `<lifecycle onDestroy>` only observes teardown, and `<show=false>` parks its nodes in a detached fragment, so a transition never runs. Stage it in state instead: keep the item rendered while marking it exiting (`class={ exiting: id === leavingId }`), then drop it from the list in a `transitionend`/`animationend` handler.

### Events

| Marko 5                                                       | Marko 6                                                                                                                  |
| ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| `this.emit("select", a, b)`                                   | call the function prop: `(input.onSelect \|\| null)?.(a, b)` — declare `onSelect?: (a: A, b: B) => void` in `Input`      |
| parent `<child on-select("handleSelect")>` + class method     | `<child onSelect=handleSelect>` or `<child onSelect(a, b) { ... }>`                                                      |
| parent `<child on-select("emit", "select")>` (re-emit chains) | pass the prop through: `<child onSelect=input.onSelect>`                                                                 |
| `this.on(...)` / `once` / `removeListener` on a component     | none — components are not emitters; use function props                                                                   |
| `this.subscribeTo(window).on("resize", fn)`                   | `<script>` + `window.addEventListener("resize", fn, { signal: $signal })`                                                |
| DOM `on-click("method", extraArg)` on native tags             | `onClick(e) { method(extraArg, e) }` — handler receives the DOM event (and in 6, custom-tag conventions add the element) |
| state-change notifications via emitted events                 | prefer the controllable pattern (`x` + `xChange`) — <https://markojs.com/docs/explanation/controllable-components.md>    |

The optional-handler idiom is `(input.onSelect || null)?.(a, b)`: handler attributes may legitimately be _falsy_, not just undefined (conditional handlers like `onClick=!clicked && handler` pass `false`), and `false?.()` still throws — `|| null` coerces any falsy value so the optional call short-circuits.

Event _naming_: Marko 5 events were dash-case strings (`on-page-change("m")`); Marko 6 props are camelCase functions (`onPageChange=`). For published components this is a breaking API rename — call it out to consumers.

### Refs & DOM access

| Marko 5                                            | Marko 6                                                                                                                                                                                            |
| -------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `key="input"` + `this.getEl("input")`              | `<input/inputEl>` … `inputEl()` (getter; browser-only)                                                                                                                                             |
| `this.getEls("item")` + `key="item[]"`             | tag variable inside the loop; iterate hoisted: `for (const el of item) ...` / `[...item]`                                                                                                          |
| `this.el`                                          | tag variable on the root element                                                                                                                                                                   |
| `this.getElId(...)` / `this.elId(...)` / `this.id` | `<id/myId>` (unique, SSR-stable string)                                                                                                                                                            |
| `this.getComponent("child")` / `getComponents`     | child exposes an API via `<return={ focus() {...} }>`; parent receives it as a tag variable: `<child/childApi>` … `childApi.focus()`. Often better: controllable props instead of imperative calls |
| `component.appendTo(el)` etc. (DOM-insert mixins)  | `Template.mount(input, node, position)`                                                                                                                                                            |
| `this.destroy()`                                   | `instance.destroy()` from `mount()`; inside templates, removal via `<if>`                                                                                                                          |
| `this.isDestroyed()`                               | none; `$signal.aborted` covers most uses                                                                                                                                                           |
| `this.rerender(...)` (legacy)                      | input-driven updates / `instance.update(input)`                                                                                                                                                    |

## Legacy widget API (marko-widgets / `@marko/compat-v4`)

No interop and no automation — rewrite by hand. Mapping for the common members:

| Widget API                                                               | Tags API                                                                                          |
| ------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------- |
| `defineComponent({ template, ... })` / `defineWidget` / `defineRenderer` | a single `.marko` file                                                                            |
| `init(config)` (browser attach)                                          | `<script>` (browser-only by construction)                                                         |
| `getInitialState(input)`                                                 | `<let>` initial values                                                                            |
| `getInitialProps(input, out)` / `getTemplateData(state, props)`          | `<const>` derivations from `input`                                                                |
| `getWidgetConfig()`                                                      | plain values passed via attributes; state that must reach the browser is serialized automatically |
| `w-bind` / `widget-types`                                                | the component boundary is the file itself                                                         |
| `w-on-*` handlers                                                        | `onEvent=` function attributes                                                                    |
| `w-id` / `this.getEl(id)`                                                | tag variables                                                                                     |
| `widgetBody` / `w-body`                                                  | `content`                                                                                         |
| `this.setProps(...)` / `this.rerender(...)`                              | input-driven updates from the parent                                                              |
| `this.state = newState` (replace)                                        | assignments to `<let>` vars                                                                       |
| jQuery `this.$()` helpers                                                | tag variable + direct DOM APIs (or the library inside `<lifecycle>`)                              |

## Template & server API

| Marko 5                                                                | Marko 6                                                                                                        | Notes                                                                                     |
| ---------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| `template.render(input)` (thenable out)                                | `template.render(input)`                                                                                       | 6 result: async-iterable + `.pipe(writable)` + `.toReadable()` + thenable + `.toString()` |
| `template.render(input, callback)`                                     | `.then()` / `await`                                                                                            | callbacks removed                                                                         |
| `template.render(input, stream)` / `template.stream(input).pipe(res)`  | `template.render(input).pipe(res)`                                                                             |                                                                                           |
| `template.renderSync(input)` → `RenderResult`                          | `template.render(input).toString()`                                                                            | throws if anything async — same contract                                                  |
| `template.renderToString(input)`                                       | `template.render(input).toString()`                                                                            |                                                                                           |
| `renderResult.appendTo(el)` / `.getComponent()`                        | `template.mount(input, node, position?)` → `{ update(input), destroy(), value }`                               | `value` = the template's `<return>`                                                       |
| `template.mount(input, el, pos)`                                       | unchanged shape                                                                                                |                                                                                           |
| `out.global` / `$global`                                               | `$global`                                                                                                      | still passed as `input.$global` at the render call                                        |
| `$global.serializedGlobals = { key: true }`                            | unchanged                                                                                                      | still the opt-in for serializing globals to the browser                                   |
| `$global.componentIdPrefix` / `widgetIdPrefix`                         | `$global.renderId`                                                                                             |                                                                                           |
| `$global.runtimeId` / CSP `cspNonce`                                   | unchanged                                                                                                      |                                                                                           |
| `out.write(...)`, `out.beginAsync()`, `out.script(...)`, `out.flush()` | removed                                                                                                        | restructure with `<await>`/templates; escalate custom streaming code                      |
| `require("marko/components").init()` browser bootstrap                 | removed                                                                                                        | bundler plugin emits hydration automatically                                              |
| `marko/node-require`                                                   | `@marko/compiler/register`                                                                                     | already the guidance in late 5                                                            |
| `marko/env` (`isDebug`)                                                | none                                                                                                           | use bundler-provided env (`process.env.NODE_ENV`)                                         |
| express `res.marko(template, input)` (`@marko/express`)                | verify the integration package's Marko 6 support; otherwise `template.render({ ...input, $global }).pipe(res)` | set `content-type` yourself when piping manually                                          |

## Taglib & package metadata

| Marko 5                                                         | Marko 6                                                                                                                                                                                                                                               |
| --------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| relative discovery from `components/`                           | from `tags/` (`tags/NAME.marko`, `tags/NAME/index.marko`, one grouping-folder level allowed)                                                                                                                                                          |
| `marko.json` `"tags-dir": "./dist/components"`                  | `marko.json` `"exports": "./dist/tags"` — for **published `node_modules` tag libraries only**; `tags-dir` still works in both majors but is discouraged, and apps need no `marko.json`                                                                |
| `marko-tag.json` attribute/nested-tag declarations              | delete; type everything via `export interface Input` (`Marko.AttrTag<T>` for attribute tags)                                                                                                                                                          |
| `marko-tag.json` compiler hook overrides (`transform`, codegen) | custom compiler work — escalate                                                                                                                                                                                                                       |
| `browser.json`                                                  | delete; use a modern bundler plugin                                                                                                                                                                                                                   |
| `marko.json` `"script-lang": "ts"`                              | unchanged — but it exists for **incremental TS adoption** (opting folders in/out); a project `tsconfig.json` already makes `.marko` files TypeScript, and published packages normally ship compiled `.d.marko` types instead of setting `script-lang` |
| adjacent `style.css` / `foo.style.css` supporting files         | unchanged                                                                                                                                                                                                                                             |

## TypeScript

| Marko 5                                                   | Marko 6                                                                                       |
| --------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| `export interface Input`                                  | unchanged                                                                                     |
| `Marko.Component<Input, State>` base class                | removed with the class                                                                        |
| `State` types                                             | become the inferred/annotated types of `<let>` values                                         |
| `renderBody?: Marko.Body`                                 | `content?: Marko.Body` (params: `Marko.Body<[Arg1, Arg2]>`)                                   |
| `Marko.Input<"button">`                                   | still exists; `Marko.HTML.Button` etc. is the common spelling for extending native tag inputs |
| `Marko.AttrTag<T>` / deprecated `Marko.RepeatableAttrTag` | `Marko.AttrTag<T>` (always iterable)                                                          |
| `Marko.Out`, `Marko.Emitter`, `Marko.RenderResult`        | removed in 6 (available from `marko@5` during interop)                                        |
| `@marko/type-check` (`mtc`) CLI                           | works for both                                                                                |

## Already deprecated in Marko 5 (clear these first)

These warn under `MARKO_DEBUG` in Marko 5 and have no future in 6 — fix them while still on 5 (Step 2 of the skill):

- `data` template variable → `input` (auto-fixed by the 5 compiler's migrate stage).
- `this.els` → `getEls(...)`; `getEl` reaching into child components; `getComponent` with repeated `key[]`; non-string `key` values.
- `RenderResult.html` → `.toString()`; `RenderResult.context` → `.out`; `getWidget`/`getWidgets` → `getComponent`/`getComponents`.
- `<await>` with a callback argument → pass a promise.
- String values for spread/dynamic attributes → objects.
- `marko/compiler` → `@marko/compiler`; `marko/node-require` → `@marko/compiler/register`.
- `browser.json` `require:` paths → `marko-dependencies:` (or better: leave `browser.json` behind entirely).
